
# Village Planning Agent 架构开发规范

> **版本**: v5.0.0
> **最后更新**: 2026-05-08
> **基于**: 模块化单体 + 领域模块聚合 + 配置驱动架构

---

## 1. 项目概述

基于 LangGraph 和 LangChain 的村庄规划智能体系统，采用**配置驱动**和**服务分层**架构，支持三层递进式规划分析。

**当前版本**: v5.0.0

---

## 2. 核心架构

### 2.1 技术栈

| 层级       | 技术            | 说明                           |
| ---------- | --------------- | ------------------------------ |
| Agent 框架 | LangGraph       | StateGraph + Send API 并行分发 |
| LLM        | ChatOpenAI 兼容 | 千问 / 智谱 GLM-4-flash 等     |
| 向量存储   | ChromaDB        | RAG 知识检索                   |
| 关系数据库 | SQLite (WAL)    | Checkpoint 持久化 + 报告存储   |
| API        | FastAPI         | SSE 流式推送                   |
| 前端       | React + Zustand | MapLibre 地图渲染              |
| 配置       | YAML + Pydantic | 维度唯一来源                   |

### 2.2 五层架构

```
用户层 (React + Zustand) ── SSE ──► API 网关 (FastAPI)
                                        │
                                   Agent 运行时 (LangGraph)
                                   ┌ conversation (路由)
                                   ├ execute_tools
                                   └ analysis (超级节点)
                                        │
                          ┌─────────────┼─────────────┐
                    modules/gis    modules/rag    services/
                    (空间分析)     (知识检索)    (报告/SSE/检查点)
```

### 2.3 三层规划架构

| 层级    | 名称     | 维度数 | 执行模式                  |
| ------- | -------- | ------ | ------------------------- |
| Layer 1 | 现状分析 | 12     | Map-Reduce 并行           |
| Layer 2 | 规划思路 | 4      | Wave 波次执行             |
| Layer 3 | 详细规划 | 10+    | Wave 波次执行，含级联依赖 |

## 3. 核心设计原则

| 原则                   | 说明                                                                     |
| ---------------------- | ------------------------------------------------------------------------ |
| **配置驱动**     | 维度、依赖、工具绑定唯一来源为 `config/phases.yaml`，变更无需改代码    |
| **状态最轻化**   | Agent 状态仅 12 字段，仅存流程指针；报告全文、GIS 数据通过服务或事件传递 |
| **服务分层**     | API 网关 → Agent 编排 → 工具调用，清晰单向依赖                         |
| **单一通信通道** | 前端仅用 SSE 长连接接收所有数据，无 REST 轮询                            |
| **工具协议统一** | 所有工具返回 `ToolResult {status, data, error}`，标准化容错            |
| **动态 RAG**     | LLM 根据当前维度摘要动态生成检索词，替代静态关键词                       |
| **级联修复自动** | 反馈注入 `human_feedback`，基于 YAML 依赖树自动计算影响范围并重跑      |
| **时光倒流**     | 报告版本化管理 + LangGraph Checkpoint，可从任意关键点恢复                |

## 4. 状态管理

### 4.1 核心状态定义（12 字段）

```python
class AgentState(TypedDict, total=False):
    messages: Annotated[List[BaseMessage], add_messages]
    session_id: str
    project_name: str

    # 进度
    phase: str                    # "init" | "layer1" | "layer2" | "layer3" | "completed"
    current_wave: int
    completed_dimensions: Dict[str, List[str]]   # {phase_id: [dim_key, ...]}

    # 报告版本索引
    report_versions: Dict[str, List[ReportVersion]]

    # 摘要索引
    summaries: Dict[str, DimensionSummary]

    # 交互
    human_feedback: Optional[str]

    # Send 分发标记
    dimension_key: Optional[str]
```

### 4.2 状态管理规范

- 状态更新采用**部分更新**模式（只返回需要更新的字段）
- 使用 `Annotated[List, add_messages]` 处理消息累加
- **强制禁止**在 State 中存储：报告全文、SSE 事件队列、GIS GeoJSON、图片数据
- 上述重量级数据交由 `ReportStore`、`SSEManager`、`GisService` 管理

## 5. 图结构设计

### 5.1 节点定义（3 物理节点）

| 节点              | 文件位置                        | 职责                                     |
| ----------------- | ------------------------------- | ---------------------------------------- |
| `conversation`  | `agent/nodes/conversation.py` | 中央路由，含反馈处理与级联修复触发       |
| `execute_tools` | `agent/nodes/tools.py`        | LLM 动态工具调用执行                     |
| `analysis`      | `agent/nodes/analysis.py`     | 维度分析超级节点（工具+RAG+LLM+SSE推送） |

### 5.2 图构建规范

```python
from langgraph.graph import StateGraph, START, END
from .state import AgentState
from .nodes.conversation import conversation
from .nodes.tools import execute_tools
from .nodes.analysis import analysis
from .routing import route_intent, check_phase_completion

def build_graph():
    builder = StateGraph(AgentState)
    builder.add_node("conversation", conversation)
    builder.add_node("execute_tools", execute_tools)
    builder.add_node("analysis", analysis)

    builder.add_edge(START, "conversation")
    builder.add_conditional_edges("conversation", route_intent, {
        "execute_tools": "execute_tools",
        "analysis": "analysis",
        END: END,
    })
    builder.add_edge("execute_tools", "conversation")
    builder.add_conditional_edges("analysis", check_phase_completion, {
        "analysis": "analysis",
        "conversation": "conversation",
        END: END,
    })
    return builder.compile()
```

### 5.3 节点函数规范

```python
async def node_function(state: AgentState, config: RunnableConfig) -> Dict[str, Any]:
    """
    节点函数说明

    Args:
        state: 当前 Agent 状态（只读）
        config: LangGraph 运行时配置

    Returns:
        部分状态更新字典
    """
    # 1. 获取配置（从 YAML）
    # 2. 调用服务层或工具
    # 3. 通过 SSE 推送事件
    # 4. 返回状态更新
    return {"field": "value"}
```

### 5.4 路由函数规范

```python
def route_intent(state: AgentState) -> Union[str, List[Send]]:
    """
    条件路由函数

    可返回:
    - str: 目标节点名或 "end"
    - List[Send]: 并行分发到 analysis 节点
    """
    if has_tool_calls(state):
        return "execute_tools"
    if should_advance(state):
        return dispatch_dimensions(state)  # 返回 List[Send]
    return END
```

## 7. 工具系统

### 7.1 工具协议（`tools/protocol.py`）

```python
from dataclasses import dataclass
from typing import Any, Optional, Protocol

@dataclass
class ToolResult:
    status: str          # "success" | "partial" | "error"
    data: Any = None
    error: Optional[str] = None

    def is_success(self) -> bool:
        return self.status == "success"

class Tool(Protocol):
    async def execute(self, context: dict) -> ToolResult:
        ...
```

### 7.2 工具注册与调用

```python
from tools.registry import ToolRegistry

# 注册
ToolRegistry.register("gis_coverage_calculator", CoverageTool())

# 调用（自动异常捕获）
result = await ToolRegistry.run("gis_coverage_calculator", {"village_name": "金田村"})
if not result.is_success():
    logger.warning(f"工具失败: {result.error}")
```

### 7.3 工具适配器

```python
# tools/adapters/langchain.py
def to_langchain_tool(tool: Tool, input_model, name, description) -> StructuredTool:
    """将 Tool 协议适配为 LangChain StructuredTool"""
```

### 7.4 工具开发规范

- 所有工具**必须**实现 `Tool` 协议
- **必须**返回 `ToolResult`，禁止裸抛异常
- 工具失败**不应**中断维度分析，调用方按 `status` 降级处理
- 新增工具后在 YAML 对应维度的 `tools` 字段中引用

## 8. RAG 知识检索

### 8.1 调用规范

```python
from modules.rag import RagService

# 动态查询生成 + 检索
query = await RagService.generate_query(dim_config, state)
docs = await RagService.search(query, top_k=3)
```

### 8.2 设计原则

- **禁止**在代码中硬编码静态检索关键词
- 检索词由 `RagService.generate_query()` 调用 LLM 根据当前维度摘要和上下文动态生成
- YAML 中的 `rag_query_hint` 仅在 LLM 生成失败时作为兜底
- 检索失败时降级为空列表，不阻断维度分析

## 9. SSE 通信规范

### 9.1 事件类型

| 事件                   | 说明                 | 携带数据                                                                 |
| ---------------------- | -------------------- | ------------------------------------------------------------------------ |
| `state_sync`         | 连接建立后的初始状态 | 完整 12 字段状态                                                         |
| `dimension_start`    | 维度开始分析         | `dimension_key`                                                        |
| `dimension_delta`    | 流式报告增量         | `dimension_key`, `content`                                           |
| `dimension_complete` | 维度分析完成         | `dimension_key`, `report_id`, `version`, `summary`, `gis_data` |
| `layer_paused`       | 层级完成等待审查     | `layer`, `reason`                                                    |
| `completed`          | 全部规划完成         | -                                                                        |

### 9.2 推送规范

```python
from services.sse import SSEManager
sse = SSEManager.get_instance()
await sse.emit(session_id, {"type": "dimension_delta", "dimension_key": dim, "content": chunk})
```

### 9.3 通信规范

- **单一通道**：前端仅通过 SSE 获取所有数据，禁止额外的状态轮询
- `dimension_complete` 必须携带 `report_id` 和 `version`，前端据此懒加载报告全文
- 每个事件必须包含 `seq` 序列号，支持断线重连

## 10. 级联修复

### 10.1 触发流程

```
用户反馈 → API 注入 human_feedback → conversation 节点检测
    → 计算影响树（基于 YAML depends_on）
    → 重置受影响维度的 completed_dimensions
    → Send 重新分发到 analysis 节点
```

### 10.2 影响计算

```python
from config.dependency import compute_impact

# 返回跨层受影响维度列表
impacted = compute_impact("land_use", "layer1")
# 例: {"layer2": ["resource_endowment"], "layer3": ["spatial_structure"]}
```

### 10.3 规范

- 级联修复**全自动**，用户只需提交反馈，无需手动指定受影响维度
- 修复产生新报告版本，旧版本保留在 `ReportStore` 中
- 版本历史通过 `report_versions` 索引追踪

## 13. 模块组织规范

```
backend/app/
├── agent/          # LangGraph 图定义、状态、路由
├── api/            # FastAPI 路由（7 个端点）
├── config/         # YAML + 加载器
├── core/           # LLM 工厂、设置、追踪
├── modules/        # 领域模块聚合
│   ├── gis/        # GIS 数据获取、分析、兜底
│   ├── rag/        # 动态查询、向量检索、知识管理
│   └── vision/     # 多模态图片分析
├── services/       # 报告存储、SSE、检查点、运行时
├── tools/          # Tool 协议、注册、适配器
└── utils/          # 日志、文本处理、上下文工具
```

### 13.1 模块间依赖方向

```
agent → modules + services + tools
modules → 独立（不相互依赖）
services → 独立或依赖 modules
tools → 依赖 modules
```

### 13.2 新增维度流程

1. 编辑 `config/phases.yaml`：添加维度 key、名称、依赖、工具、提示词
2. 如需新工具：实现 `Tool` 协议并注册
3. **无需修改任何 Python 编排代码**

## 14. 错误处理规范

```python
from utils.logger import get_logger

logger = get_logger(__name__)

# 工具错误：返回 ToolResult
async def execute(self, context) -> ToolResult:
    try:
        # ...
        return ToolResult(status="success", data=result)
    except Exception as e:
        logger.error(f"工具失败: {e}")
        return ToolResult(status="error", error=str(e))

# LLM 错误：降级或重试
try:
    response = await llm.ainvoke(prompt)
except Exception as e:
    logger.warning(f"LLM 调用失败，使用降级策略: {e}")
    response = fallback_response()

# 服务错误：记录并降级
try:
    docs = await RagService.search(query)
except Exception:
    docs = []  # 降级为空知识，不影响报告生成
```

## 15. 性能优化原则

- **状态轻量**：State 仅 12 字段，减少 Checkpoint 序列化开销
- **并行工具执行**：`asyncio.gather` 并发调用工具和 RAG
- **SSE 直推**：事件不经过 State 累积，直接推送前端
- **报告按需加载**：SSE 只推摘要，全文通过 `report_id` 懒加载
- **动态波次**：依赖自动计算 Wave，同波次内并行执行

---

## 16. 禁止事项

- ❌ 在 State 中存储报告全文、GIS GeoJSON、图片二进制
- ❌ 在 Python 代码中硬编码维度元数据（依赖、工具绑定）
- ❌ 工具中裸抛异常（必须返回 `ToolResult`）
- ❌ 在路由函数中执行耗时操作
- ❌ 使用阻塞式 I/O（使用异步）
- ❌ 在循环中直接调用 LLM（需并行化）
- ❌ 使用静态关键词作为 RAG 查询（必须动态生成）
- ❌ 手动指定级联修复的目标维度（应由系统自动计算）

---

## 17. 微服务预留

当前为**模块化单体**，但以下设计支持未来平稳拆分：

- `modules/gis`、`modules/rag` 仅通过接口对外暴露
- `services/` 中的 `RuntimeService` 封装了 Graph 执行入口
- `tools/adapters/` 提供远程调用适配器模板
- 拆分时仅需将 Python import 替换为 gRPC/HTTP 客户端，Agent 节点代码无需修改
